# CC:Tweaked складской терминал-маркетплейс (Create Stock Ticker) — Дизайн

Дата: 2026-06-18
Статус: одобрен к написанию плана

## Context

На сервере (Create 6, MC 1.21.1) есть главный склад **Core**: все ресурсы свозятся туда,
вся база соединена в общую пакетную сеть Create. Точка доступа ко всей сети — блок
**Stock Ticker**, который CC:Tweaked видит как периферию `Create_StockTicker`
(native-интеграция Create 6). Крейтовский интерфейс заказа неудобен; нужен свой
**сенсорный терминал-маркетплейс** под свои названия: смотреть остатки всей базы, искать
предмет, заказывать N штук с доставкой на адрес пакетной сети.

Это **под-проект SP1 (MVP)** более крупной идеи. Отложено явно:
- **SP2** — DPS-трекер поступления ресурсов (что автоматизировано, что нет).
- **SP3** — цепочка крафта + подсветка нехватки. Невозможна на чистом Create+CC
  (у CC нет реестра рецептов); потребует ручной базы рецептов. Отдельный спек.

## Подтверждённые факты (проверено in-game + офиц. вики Create)

Периферия `Create_StockTicker`, методы (`peripheral.getMethods`):
`stock`, `list`, `getItemDetail`, `getStockItemDetail`, `requestFiltered`.

- `stock([detailed])` → non-sparse список (ipairs). База: `{name, count, nbt}`.
  При `detailed=true` добавляет `displayName, maxCount, itemGroups, tags, enchantments`.
  В сети тестово **340 позиций**; базовая запись — `{name="minecraft:oxeye_daisy", count=1}`.
- `requestFiltered(address, filter1, ...)` → число заказанных. `address` (string) —
  адрес назначения пакетной сети. Фильтр: `{name="minecraft:apple", _requestCount=10}`.
  Доп. ключи фильтра: `_mode` (`contains`/`exact`/`contained`), `_op`
  (`>`,`>=`,`<`,`<=`,`==`,`~=`,`any`,`all`,`glob`,`regex`,`type`,`not`).
- API **не перечисляет адреса** — список адресов задаёт пользователь (см. ниже).
- **Display-имён в базовом `stock()` нет**; кастомные ярлыки храним сами.

Адреса доставки (известные, расширяемо):
- **`Main`** — сундук у склада, выдача в руки.
- **`Core`** — общий склад.
Перед заказом UI даёт выбрать целевой адрес; список — из `addresses.cfg`.

Железо (стороны не важны — авто-детект по типу):
- монитор: тип `monitor_advanced` (или `monitor`) — тач + цвет;
- `Create_StockTicker` — данные + заказ;
- склад/`create:item_vault` — **в MVP не используется** (заказ идёт на адрес `Main`).

## Ограничения / решения

- **Clipboard не читается.** Create Clipboard — это item, не периферия; CC его содержимое
  не видит. Кастомные имена и адреса держим в **редактируемом файле** на компе
  (`names.cfg`, `addresses.cfg`). Авто-импорт из Clipboard — отложен (SP-later, если в игре
  найдётся отдающий метод).
- **На тач-мониторе нет клавиатуры.** Навигация — по категориям (тапы). Текстовый поиск —
  поле, которое по тапу берёт фокус и принимает символы с **физической клавиатуры компа**
  (`char`/`key`), живой фильтр. Экранную клавиатуру не рисуем.
- **Мультиадрес доставки.** Список из `addresses.cfg` (старт: `Main`, `Core`). UI даёт
  выбрать адрес перед подтверждением заказа; дефолт — первый в списке.

## Архитектура

Один Advanced Computer. Периферии находятся по типу (`peripheral.find`), стороны не
хардкодятся; конфиг допускает явный override. Два параллельных цикла через
`parallel.waitForAll`:
- **refresh-цикл** — таймер `REFRESH=2`с: `stock.fetch()` → модель → `ui.render()`;
- **input-цикл** — `os.pullEvent`: `monitor_touch` / `char` / `key`.

## Компоненты (файлы)

Каждый модуль — одна ответственность, чистые функции тестируемы изолированно;
`main()` запускается, только если файл — программа (не `require`).

- `config.lua` — `REFRESH=2`, цвета/раскладка, опц. override имён периферий. Адреса —
  отдельный `addresses.cfg`.
- `peripherals.lua` — `findTicker()`, `findMonitor()` (типы `monitor_advanced`/`monitor`);
  понятная ошибка + стоп, если не найдено.
- `stock.lua` — `fetch(ticker)`: `ticker.stock(true)` → нормализ. список
  `{id, count, display, group}`, сортировка; кэш прошлого снимка (задел под SP2 DPS).
- `names.lua` — загрузка/сохранение карты `id→ярлык` из `names.cfg`; автозасев из
  `displayName`; `label(id)` (fallback: displayName → красиво-из-id).
- `addresses.lua` — загрузка списка адресов из `addresses.cfg` (старт: `Main`, `Core`);
  `list()`, `default()`; дозапись новых адресов вручную в файл.
- `ui.lua` — рендер (левая колонка категорий + сетка + поле поиска + кейпад количества
  + выбор адреса доставки), хит-тест, состояние (категория / строка фильтра / фокус поиска /
  страница / выбор / выбранный адрес); чистые `filter()`, `inside()`, `layout()` — юнит-тестируемы.
- `order.lua` — `place(ticker, id, qty, address)` → `ticker.requestFiltered(address, {name=id,
  _requestCount=qty})`; возврат числа → тост.
- `startup.lua` — обвязка + загрузка конфигов + `parallel.waitForAll(refresh, input)`.

## Поток данных

1. Старт: найти периферии → загрузить `config`, `names.cfg`, `addresses.cfg` → первый
   `stock.fetch` → render. Адрес по умолчанию — первый в списке.
2. refresh каждые 2с: новый снимок стока → пересорт/перегруппировка → render
   (сохранять текущие категорию/фильтр/выбор/адрес).
3. Ввод:
   - тап по категории → сменить активную группу;
   - тап по полю поиска → фокус; `char`/`key` → правка фильтра → render;
   - тап по селектору адреса → переключить целевой адрес (`Main`/`Core`/…);
   - тап по предмету → открыть кейпад количества;
   - кейпад подтвердить → `order.place(…, address)` → тост «Заказано N×ярлык → <адрес>».

## Обработка ошибок

- Нет тикера/монитора при старте → сообщение в терминал, стоп.
- `stock()` упал/пусто → «сеть недоступна», UI не падает, ретрай на след. тике.
- `requestFiltered` вернул `0` → тост «нет в наличии / не отправлено».
- Кол-во > наличия → зажать до `count`; мусорный ввод → игнор.

## Тестирование (оба сетапа)

- **lua5.1 + мок-харнес** (`test/mock-cc.lua` + `test/spec.lua`): стаб `Create_StockTicker`
  (`stock` отдаёт фейк-список; `requestFiltered` логирует аргументы), стаб
  `monitor`/`term`, скриптуемая очередь событий. Юнит-кейсы: нормализация стока,
  группировка по категориям, фильтр поиска, карта имён + fallback, хит-тест кейпада/сетки,
  сборка аргументов `requestFiltered`, зажатие количества.
  Запуск: `lua5.1 test/spec.lua` → `ALL PASS`.
- **CraftOS-PC headless**: smoke — грузится без рантайм-ошибок, реальные
  `term/monitor/parallel/os` не нарушены (тикер инжектится шим-стабом, периферии которого
  CraftOS не имеет).
- **In-game (пользователь)**: продвинутый монитор слева, тикер, заказ 1× дешёвого предмета
  на `Main` → посылка приходит в сундук Main; остатки авто-обновляются ~2с.

## Инфраструктура окружения

- Lua и CC-эмулятора в окружении не было — ставятся: `lua5.1` (apt, нужен пароль sudo —
  ставит пользователь, либо локальная сборка Lua 5.1.5 в `~/.local` без sudo) + CraftOS-PC
  (snap/AppImage/PPA).
- Деплой в игру: `pastebin get` / `wget` из репо, либо копия в
  `world/computercraft/computer/<id>/`.
